# Daftar simbol & konstruk Regex (PCRE-modern, lintas-dialect) — tabel komprehensif

Di bawah ini adalah daftar komprehensif simbol, escape, kelas, kuantor, grup, assertion, modifier, dan konstruk-konstruk khusus yang berlaku untuk *PCRE / PCRE2* dan penerapan modern lain (Perl, Java, .NET, Python, JavaScript, alat CLI seperti `grep -P` / `pcregrep` / `rg --pcre2`, serta catatan perbedaan pada `sed`/`awk`). Setiap baris menjelaskan **nama**, **kegunaan**, **sintaks / contoh**, **jenis**, **catatan penting (perbedaan antar engine / kemampuan)** dan referensi dokumentasi utama di bagian sumber paling bawah.

> Catatan singkat: banyak konstruk inti (metacharacters, kelas, kuantor, lookarounds, named groups, backreferences, modifiers inline) dideskripsikan lengkap di dokumentasi PCRE2; bahasa/libraries lain meniru sebagian besar tetapi kadang menolak fitur tertentu (mis. variable-length lookbehind, recursion, subroutine calls, balancing groups). Sumber utama: dokumentasi PCRE2, Perl, MDN (JS), Python `re`, Java `Pattern`, .NET, dokumentasi alat CLI (grep/pcregrep/ripgrep). ([pcre.org][1])

## Catatan penting per-kenyataan implementasi & perbedaan lintas-engine (ringkas, inti)

1. **PCRE / PCRE2** adalah referensi fungsional paling lengkap untuk sintaks yang sering disebut “Perl-compatible”. Banyak konstruk lanjutan (atomic groups, possessive quantifiers, subroutine calls, callouts, `(*VERB)` family, branch reset, variable-length lookbehind with limit, dll.) terdokumentasi di PCRE2. Untuk sebagian besar simbol di atas, rujukan utama adalah dokumentasi PCRE2. ([pcre.org][1])

2. **Perl** memiliki sintaks yang paling kaya (dan sering menjadi dasar PCRE). Perl terus memperkenalkan fitur; beberapa fitur eksperimental muncul (mis. variable-length lookbehind eksperimental di versi modern). Perbedaan kecil pada perilaku exist. ([perldoc.perl.org][4])

3. **JavaScript** (ECMAScript) meniru banyak sintaks Perl tetapi **tidak** mendukung beberapa fitur PCRE (mis. subroutine recursion `(?R)`, possessive quantifiers, `\K`, `(*VERB)`), walaupun JS modern menambahkan lookbehind and Unicode property escapes (`\p{...}`) pada spesifikasi terbaru (dengan flag `u`). Periksa MDN untuk detail versi. ([developer.mozilla.org][3])

4. **Python** `re` standard mengikuti gaya Perl namun **membatasi lookbehind ke fixed length** (pattern di lookbehind harus memiliki panjang tetap). Untuk fitur lebih lengkap (variable-length lookbehind, subroutine, overlapped matches, dll.) gunakan modul pihak ketiga `regex`. ([Python documentation][5])

5. **Java** (`java.util.regex.Pattern`) kompatibel dengan Unicode properties (`\p{…}`) dan sejak Java 9 memperluas dukungan lookbehind; possessive quantifiers tersedia. Namun tidak semua konstruk PCRE eksotis tersedia. ([docs.oracle.com][6])

6. **.NET** memiliki fitur unik seperti *balancing groups* `(?<A-B>...)` untuk matching nested constructs; ini adalah alternatif bagi recursion/subroutine di engine lain. .NET juga mendukung named groups, inline options, dan fitur luas lainnya. ([Microsoft Learn][7])

7. **Alat CLI**:

   * `grep` (GNU grep) mendukung BRE/ERE; PCRE (PCRE2) hanya bila tersedia (option `-P` untuk PCRE2 on some builds) dan dokumentasinya merujuk pada `pcre2syntax`. `pcregrep`/`pcre2grep` adalah utilitas khusus PCRE. ([gnu.org][8])
   * `sed` / `awk` umumnya memakai POSIX BRE/ERE — fitur PCRE lanjutan umumnya **tidak** tersedia kecuali tool yang menggunakan PCRE. ([gnu.org][9])
   * `ripgrep (rg)` memiliki *optional* PCRE2 support via build feature; gunakan `-P/--pcre2`. Jika build tanpa opsi, fitur lookaround/backrefs mungkin tidak tersedia. ([GitHub][10])

8. **Lookbehind variable-length**: PCRE2 menyediakan *length-limited variable-length lookbehind* dengan batas default (mis. 255) yang dapat diatur pada build/compile-time; Perl/Java/.NET/JS juga memiliki variasi dukungan. Python `re` tidak mendukung variable-length lookbehind, sedangkan modul `regex` di Python mendukungnya. Sangat penting untuk memeriksa engine saat menggunakan lookbehind dengan kuantor. ([PCRE2 Project][11])

9. **Performa & ReDoS**: beberapa pattern (terutama yang menempatkan banyak alternatif dengan kuantor greedylah) rentan ReDoS (catastrophic backtracking). Gunakan atomic groups `(?>...)`, possessive quantifiers, atau refactor pattern untuk menghindari backtracking berlebih. Ini adalah pertimbangan praktis penting ketika merancang pattern untuk public reference. ([pcre.org][1])

## Saran struktur referensi publik (untuk pintasan Anda)

Untuk membuat referensi publik yang bisa dipakai lintas-bahasa, rekomendasi struktur halaman/CSV/markdown:

1. **Kolom**: simbol, nama, contoh sintaks, tipe (metachar/class/assertion/quantifier/group/modifier), dukungan engine (PCRE/Perl/JS/Python/Java/.NET/grep/sed/awk/rg), catatan kompatibilitas, contoh penggunaan, link dokumentasi per-engine.
2. **Halaman per kategori**: metacharacters, character classes, escapes, quantifiers, groups & backrefs, lookarounds, recursion/subroutines, modifiers, engine-specific features.
3. **Tes kompatibilitas**: sertakan matrix singkat (✔/✖/partial) untuk fitur kunci per engine dengan tautan ke dokumentasi engine.
4. **Contoh nyata**: berikan contoh valid di masing-masing engine (kapan perlu escape ekstra atau fitur tidak tersedia, sertakan alternatif e.g. `\K` vs capturing workaround).
5. **Performa**: tip & contoh anti-ReDoS (atomic groups, possessive quantifiers, menulis ulang pola).
